## Part 1: Testing RAG Document Chat (Claude)

This validates the core RAG pipeline — uploading a document and chatting with it using Claude as the response generator.

### Step 1: Environment Setup

1. Confirm environment variables are set:
   - `ANTHROPIC_API_KEY` — Claude (document processing, response generation, self-eval)
   - `OPENAI_API_KEY` — OpenAI (embedding generation for semantic search)
   - `SUPABASE_URL` and `SUPABASE_SERVICE_ROLE_KEY`
2. Start dev server:
   ```bash
   cd "c:/Users/james/Master/BrightHub/BRun/v4-show" && npm run dev
   ```
3. Open browser: `http://localhost:3000`
4. Sign in to your account

### Step 2: Create Knowledge Base & Upload Document

1. Navigate to `/dashboard` → click **"RAG Frontier"** (teal button with book icon)
2. Click **"Create Knowledge Base"**
   - Name: `E13 Test KB`
   - Description: `Testing RAG and LoRA integration`
3. Click **"Create"**, then select the new knowledge base
4. In the **"Upload Document"** panel:
   - Select a PDF, DOCX, TXT, or MD file (under 50 MB)
   - Optionally add a description
   - Click **"Upload"**
5. Watch the status progression:
   - `uploading` → `processing` (1-5 minutes — Claude extracts sections, facts, entities)

### Step 3: Answer Expert Questions & Verify

1. When status reaches `awaiting_questions`:
   - Click the document → go to **"Expert Q&A"** tab
   - Answer 3-8 questions generated by Claude
   - Click **"Submit Answer"** for each, or **"Skip"** if not applicable
2. After all questions are handled, click **"Verify Document"**
3. Status changes to `ready` — the document is now available for chat

### Step 4: Chat with Document (RAG Only Mode)

1. Go to the **"Chat"** tab on the document detail page
2. Mode selector should show **"RAG Only"** selected (default)
3. Type a question about the uploaded document content
4. Press **Enter** to submit
5. **Expected behavior:**
   - System generates a HyDE (hypothetical answer) for better embedding search
   - Retrieves relevant sections and facts via pgvector similarity search
   - Claude generates a cited response from the assembled context
   - Citations appear below the response (click to expand)
   - Self-eval score appears (percentage + pass/fail checkmark)
   - Response time displayed
6. Click **"Evaluate"** on any response to run the full 5-metric quality evaluation

### Step 5: Verify Quality Dashboard

1. After chatting, navigate to the document detail page
2. Click the **"Quality"** button (top-right area)
3. **Expected metrics displayed:**
   - Faithfulness — Is the answer supported by the retrieved context?
   - Answer Relevance — Does the answer address the question?
   - Context Relevance — Were the retrieved sections/facts relevant?
   - Answer Completeness — Is the answer thorough?
   - Citation Accuracy — Do citations match the actual context?
4. View the composite score and score trends

### RAG-Only Test Checklist

- [ ] Knowledge base created successfully
- [ ] Document uploaded and status reaches `processing`
- [ ] Expert questions generated and answerable
- [ ] Document verified and status is `ready`
- [ ] RAG chat returns cited, document-grounded answers
- [ ] Self-eval scores appear on responses
- [ ] Quality dashboard shows metrics

## Part 2: Testing LoRA + RAG Integration (E12 Features)

This validates the new E12 functionality — using your fine-tuned LoRA models to generate responses, either standalone or with RAG-retrieved context.

### Prerequisites

You need a deployed LoRA adapter on RunPod. If you don't have one:

1. **Train a model**: Go to `/datasets` → create/select a dataset → go to `/pipeline/configure` → start a training job
2. **Wait for completion**: Monitor at `/pipeline/jobs` → click the job → wait for `completed` status
3. **Deploy the adapter**: On the completed job's page, click **"Deploy to Endpoints"**
   - System creates two RunPod endpoints: **Control** (base model) and **Adapted** (base + LoRA)
   - Wait 2-5 minutes for endpoints to show `ready` status
4. **Confirm environment variables** (in Vercel Dashboard or `.env.local`):
   - `INFERENCE_API_URL` — Control endpoint proxy URL
   - `INFERENCE_API_URL_ADAPTED` — Adapted endpoint proxy URL
   - `INFERENCE_MODE` — Set to `pods` if using RunPod Pods, or `serverless` if using RunPod Serverless

> **Current LLM**: The system uses **Mistral 7B Instruct v0.2** (`mistralai/mistral-7b-instruct-v0.2`) as the default base model for LoRA fine-tuning and inference. Your LoRA adapters are applied on top of this model.

### Step 1: Test LoRA Only Mode

This mode skips RAG retrieval entirely — your query goes directly to the LoRA model without any document context.

1. Navigate to a document's **"Chat"** tab (any document — it won't be used)
2. Click the mode selector and choose **"LoRA Only"**
3. **Verify: Model selector dropdown appears** below the mode selector
   - Shows a `<Cpu>` icon label "LoRA Model"
   - If you have deployed models: dropdown lists them with format `{job_name} — {base_model} ({dataset_name})`
   - If no deployed models: shows orange text "No deployed models found. Train and deploy a LoRA adapter first."
4. Select your deployed model from the dropdown
5. Type a general question (not necessarily about the document)
6. Press **Enter**
7. **Expected behavior:**
   - Query is sent directly to your LoRA endpoint via `callInferenceEndpoint()`
   - Response comes from the Mistral 7B model with your LoRA adapter applied
   - No citations (LoRA doesn't have document context)
   - Self-eval shows `null` (no context to evaluate against)
   - Response time reflects the LoRA inference latency (typically 2-15 seconds)

### Step 2: Test RAG + LoRA Mode

This mode retrieves document context via the RAG pipeline, then sends that context to your LoRA model instead of Claude.

1. Select a document that is in `ready` status (has been processed and verified)
2. Go to **"Chat"** tab
3. Select **"RAG + LoRA"** mode
4. **Verify: Model selector dropdown appears**
5. Select your deployed model
6. Type a question **about the document's content**
7. Press **Enter**
8. **Expected behavior:**
   - HyDE hypothetical answer generated (by Claude)
   - Context retrieved from pgvector (sections + facts)
   - Assembled context sent to your LoRA model as a prompt with the format:
     ```
     Context from knowledge base:
     {assembled_context}
     ---
     Question: {your question}
     Answer using only the context above:
     ```
   - Response generated by your LoRA/Mistral model
   - Citations parsed best-effort (looks for `[Section: title]` or `[Fact: content]` patterns)
   - Self-eval runs via Claude (evaluates LoRA's answer against the context)
   - Self-eval score may be lower than RAG-only, since LoRA may not follow context as tightly as Claude

### Step 3: Compare RAG Only vs RAG + LoRA

1. On the same document, ask the **same question** in both modes:
   - First in **"RAG Only"** (Claude generates response)
   - Then in **"RAG + LoRA"** (your LoRA model generates response)
2. Compare:
   - Response quality and style
   - Citation coverage (Claude will have more structured citations)
   - Self-eval scores (Claude typically scores higher on faithfulness)
   - Response time (LoRA may be faster or slower depending on pod configuration)

### Step 4: Test Error Handling

1. **No model selected**: Select "LoRA Only" mode → **don't** select a model → try to submit
   - **Expected**: Submit button is disabled (cannot click)
   - **Expected**: If bypassed, toast error "Please select a LoRA model before querying."

2. **Pod not running**: Select a model whose RunPod pod is currently stopped → submit a query
   - **Expected**: Error message in response: "Cannot connect to the LoRA model endpoint. The pod may be stopped."

3. **Model endpoint terminated**: Select a model that was previously deployed but endpoint was deleted
   - **Expected**: Error message: "The selected LoRA model endpoint is not available. It may have been terminated or is not ready."

4. **Switch modes**: Toggle between RAG Only → LoRA Only → RAG + LoRA → RAG Only
   - **Expected**: Model selector appears/disappears correctly
   - **Expected**: When switching back to RAG Only, model selection is cleared

### LoRA + RAG Integration Test Checklist

- [ ] LoRA Only mode: model dropdown appears when selected
- [ ] LoRA Only mode: query sends directly to LoRA endpoint (no RAG retrieval)
- [ ] LoRA Only mode: response comes from LoRA model (different style than Claude)
- [ ] RAG + LoRA mode: model dropdown appears
- [ ] RAG + LoRA mode: context retrieved before sending to LoRA
- [ ] RAG + LoRA mode: self-eval runs on LoRA's response
- [ ] Submit disabled when LoRA mode selected but no model chosen
- [ ] "No deployed models" message shown when none exist
- [ ] Error messages are actionable when pod is down
- [ ] Mode switching clears model selection correctly

## Part 3: Testing with Serverless Endpoints (Mistral LLM)

If you're using RunPod **Serverless** instead of **Pods**, the flow is the same but the configuration differs:

### Serverless Configuration

1. Set `INFERENCE_MODE=serverless` in environment variables
2. The serverless endpoints use the RunPod Serverless API format (submit job → poll status → get result)
3. Your adapter is loaded at runtime via the `LORA_MODULES` environment variable on the worker

### Key Differences from Pods

| Aspect | Pods | Serverless |
|--------|------|------------|
| Cold start | Pod always running | Worker spins up on demand (30-120s cold start) |
| Cost | Per-hour billing | Per-second billing (cheaper for intermittent use) |
| Adapter loading | Adapter pre-loaded on startup | Adapter loaded per-request from signed URL |
| API format | OpenAI-compatible `/v1/chat/completions` | RunPod wrapper submit/status/output |
| Environment var | `INFERENCE_API_URL_ADAPTED` | `RUNPOD_API_KEY` + `RUNPOD_ENDPOINT_ID` |

### Testing Serverless + LoRA in RAG

1. Confirm serverless endpoint is active:
   ```bash
   curl -H "Authorization: Bearer $RUNPOD_API_KEY" \
     "https://api.runpod.ai/v2/${ENDPOINT_ID}/health"
   ```
2. Follow the same test steps as Part 2 above
3. **Expect longer response times** due to cold starts (first request may take 60-120s)
4. After first request, subsequent requests should be 5-20s while the worker is warm

### Serverless Debugging

- **Timeout errors**: Serverless workers have a 5-minute timeout by default. Long contexts may exceed this.
- **`EngineDeadError`**: Set `DISTRIBUTED_EXECUTOR_BACKEND=mp` on the serverless endpoint configuration.
- **Empty response**: Check that `max_tokens` is set appropriately in the request. Default is 512 tokens.
- **Adapter not loading**: Verify the adapter path in Supabase Storage matches what's in `pipeline_inference_endpoints.adapter_path`.

## Part 4: Missing Features — Descriptions

The following features are not yet implemented but would enhance the RAG Frontier module:

### 3. Knowledge Base Sharing

**What it is**: Allow users to share knowledge bases with other team members. Currently, all knowledge bases are private to the creating user.

**Why it matters**: In team environments, multiple analysts may need to query the same set of documents. Without sharing, each user must upload the same documents, wasting storage and processing time.

**What implementation requires**:
- A `rag_knowledge_base_permissions` table with `knowledge_base_id`, `user_id`, `role` (viewer/editor/admin)
- UI in the knowledge base settings to invite users by email
- Row-level security (RLS) policies updated to check the permissions table
- Service layer changes to filter knowledge bases by both ownership and shared access
- Shareable link generation (optional)

**Complexity**: Medium — DB schema + RLS + UI + service layer changes

---

### 4. Bulk Document Upload

**What it is**: Upload multiple documents to a knowledge base in a single action, rather than one-by-one.

**Why it matters**: Enterprise users often have 10-100+ documents to add to a knowledge base. The current one-at-a-time flow is tedious and error-prone.

**What implementation requires**:
- Update `DocumentUploader.tsx` to accept multiple files (change `<input>` to `multiple`)
- A batch upload API endpoint that queues N documents for sequential processing
- Progress tracking UI showing per-document status (uploading/processing/ready/error)
- Background processing queue with error isolation (one failed document shouldn't block others)
- Aggregate status on the knowledge base (e.g., "12/15 documents ready")

**Complexity**: Medium — Upload UI is simple, but reliable background queue processing is the hard part

---

### 5. Document Versioning

**What it is**: Upload a new version of an existing document while preserving the previous version's data.

**Why it matters**: Documents evolve — policies get updated, research papers are revised, manuals get new editions. Currently you must delete the old document (losing all chat history and embeddings) and re-upload.

**What implementation requires**:
- Add `version` tracking to `rag_documents` (already has a `version` column set to 1)
- Archive endpoint: mark old version as `archived` instead of deleting
- New upload flow: "Replace Document" option that creates a new version linked to the same document group
- Re-index sections, facts, and embeddings for the new version
- Version history UI to compare or switch between versions
- Migration of existing chat history references

**Complexity**: High — Requires careful handling of embedding re-indexing and chat history migration

---

### 6. Cross-Document Search (Backend Exists, UI Missing)

**What it is**: Query across all documents in a knowledge base simultaneously, rather than one document at a time.

**Why it matters**: The real power of a knowledge base emerges when you can ask questions that span multiple documents. "Compare the Q1 and Q2 reports" requires cross-document context.

**Current state**: The `queryRAG()` backend already accepts `knowledgeBaseId` without `documentId`. Embeddings are stored per-document but can be searched knowledge-base-wide. The only gap is UI.

**What implementation requires**:
- "Chat with Knowledge Base" button on the knowledge base detail page (not just individual documents)
- Updated `RAGChat` component to work without a `documentId`
- Citation display showing which document each citation came from (currently assumes single document)
- Knowledge base-level chat history view

**Complexity**: Low — Backend is ready, just needs a UI entry point and citation attribution

---

### 7. Export Chat History

**What it is**: Download chat transcripts as structured files (JSON, PDF, or Markdown).

**Why it matters**: Users need to reference, share, or archive Q&A sessions. Researchers may need citations for publications. Compliance teams may need audit trails.

**What implementation requires**:
- Export button in the `RAGChat` component header
- API endpoint `GET /api/rag/query/export?documentId=...&format=json|md|pdf`
- JSON format: structured data including query, response, citations, timestamps, quality scores
- Markdown format: human-readable transcript with formatted citations
- PDF format: polished document with headers, citations, and quality metrics (requires PDF generation library)

**Complexity**: Low-Medium — JSON/MD are straightforward; PDF requires a rendering library

---

### 8. Auto-Verify Documents

**What it is**: Option to skip the Expert Q&A verification step and immediately mark documents as `ready`.

**Why it matters**: The Expert Q&A step improves knowledge grounding but adds friction. For trusted internal documents (company policies, product docs, research papers with known accuracy), the verification step adds delay without proportional value.

**What implementation requires**:
- "Skip Expert Verification" checkbox in the document upload flow
- A `fast_mode` field already exists in `rag_documents` — extend this to optionally skip the Q&A step
- Status transition: `processing` → `ready` (skipping `awaiting_questions`)
- Processing service update: when `fast_mode` + `skip_verification`, don't generate expert questions
- Quality warning badge on auto-verified documents in the UI

**Complexity**: Low — Most infrastructure already exists via `fast_mode`

---

### 9. Semantic Document Search UI

**What it is**: A search interface to find documents by semantic meaning, not just filename.

**Why it matters**: As knowledge bases grow to dozens or hundreds of documents, finding the right one by name becomes impossible. Users need to search by concept: "documents about revenue forecasting" rather than remembering `Q1-Financial-Report-2025.pdf`.

**What implementation requires**:
- Search bar component at the top of the knowledge base document list
- Backend: embed the search query, run pgvector similarity search against document-level embeddings (Tier 1)
- Results: ranked list of documents with similarity scores and matching snippet previews
- Incremental search (debounced, search-as-you-type)
- The embedding infrastructure already supports Tier 1 (document-level) embeddings

**Complexity**: Low-Medium — Embedding search exists, just needs a UI wrapper and document-level search endpoint

---

### 10. Deployment Status in Models Page

**What it is**: Show which trained LoRA models have active RunPod endpoints in the models/jobs list view.

**Why it matters**: Currently, users navigate to individual job pages to discover if a model is deployed. There's no at-a-glance view of deployment status across all jobs.

**What implementation requires**:
- Join `pipeline_training_jobs` with `pipeline_inference_endpoints` in the jobs list API
- Add a "Deployed" badge (green dot + endpoint type) on the job card in `/pipeline/jobs`
- "Deploy" / "Terminate" quick actions directly from the list view
- Link to test endpoint from the job card
- Status auto-refresh when endpoints are transitioning

**Complexity**: Low — Database join is simple, just needs UI badges and a modified list query

## Part 5: CEO Perspective — Highest-Value Features for Enterprise Customers

*Thinking as the CEO of BrightRun, a frontier RAG + LoRA integration platform, evaluating what our highest-value enterprise customers are asking for and where the product should go next.*

---

### The Customer Feedback Signal

Our highest-value customers fall into three segments:

1. **Enterprise Knowledge Teams** — Companies with 500+ internal documents (policies, SOPs, technical docs) who want AI-powered search and Q&A across their corpus
2. **AI/ML Teams** — Organizations fine-tuning models on proprietary data who want to combine domain expertise (LoRA) with factual grounding (RAG)
3. **Content & Research Organizations** — Analysts, researchers, and content teams who generate insights from document collections

Each segment has distinct pain points, but the feedback converges on several high-signal themes:

---

### Tier 1: Revenue-Critical Features (Build Next)

#### 1. Cross-Document Knowledge Base Chat 🔥

**Customer signal**: "I uploaded 30 documents to my knowledge base. Why can I only chat with them one at a time?"

**Revenue impact**: **Very High** — This is the single most requested feature. It transforms the product from a "document chatbot" into a "knowledge engine." The backend already supports it; this is primarily a UI unlock.

**Recommended priority**: Immediate (1-2 days of work)

**Why it's highest value**: The marginal cost is near-zero (backend is done) but the perceived value leap is enormous. Every enterprise demo is limited by this gap.

---

#### 2. Auto-Verify with Selective Q&A

**Customer signal**: "I trust these documents. I don't want to answer 8 questions per document when I'm uploading 50 files."

**Revenue impact**: **High** — Reduces onboarding friction from hours to minutes for bulk operations. The `fast_mode` infrastructure already exists.

**Recommended priority**: Immediate (half-day of work)

**Why it's highest value**: This is the single biggest friction point in user onboarding. Every customer who uploads more than 3 documents hits this wall. The fix is minimal.

---

#### 3. Bulk Document Upload with Progress Tracking

**Customer signal**: "I have 200 PDFs. I can't sit here and upload them one by one."

**Revenue impact**: **High** — Directly blocks enterprise adoption. No serious customer will use the product at scale without this.

**Recommended priority**: This week (2-3 days of work)

**Why it's highest value**: Combines with auto-verify to create a viable "enterprise onboarding" flow. Upload 200 docs → auto-verify → chat with entire knowledge base within an hour.

---

### Tier 2: Differentiation Features (Build This Quarter)

#### 4. Comparative Analysis Mode

**Customer signal**: "I need to compare what Document A says about pricing vs what Document B says. Make it easy."

**What it is**: A dedicated comparison interface where users select 2-4 documents and ask comparison questions. The system retrieves context from each document and presents side-by-side answers with source attribution.

**Revenue impact**: **High** — This is a genuinely novel feature that competitors don't have. It leverages our multi-document embedding search in a way that creates immediately visible business value for analysts and researchers.

**Implementation complexity**: Medium — Requires a new UI mode, modified context assembly (tag sections by document source), and structured comparison prompt engineering.

---

#### 5. RAG Quality Analytics & A/B Testing Dashboard

**Customer signal**: "How do I know if my LoRA model is actually better than Claude for my documents?"

**What it is**: A dashboard that lets users run the same set of questions through both RAG-only (Claude) and RAG+LoRA, then see comparative quality scores, response latency, and citation accuracy side by side. Essentially A/B testing for AI response quality.

**Revenue impact**: **High** — This is the feature that justifies the premium tier. Enterprise ML teams need quantitative evidence that their fine-tuning investment is paying off. This metric makes LoRA ROI measurable.

**Implementation complexity**: Medium — Quality metrics infrastructure exists (E09/E10). Needs a comparison test runner and visualization.

---

#### 6. Knowledge Base Sharing with Role-Based Access

**Customer signal**: "My team of 8 analysts all need access to the same knowledge base."

**Revenue impact**: **Medium-High** — Blocks team adoption (currently each person needs their own copy). This is table stakes for any enterprise product.

**Recommended priority**: This quarter, after Tier 1 features

---

### Tier 3: Moat-Building Features (Strategic Investment)

#### 7. Automated Knowledge Freshness & Re-Indexing

**What it is**: Let users point a knowledge base at a folder (Dropbox, Google Drive, SharePoint, S3) and automatically re-index when documents change. Version tracking built in.

**Why it's strategic**: This transforms the product from a point-in-time tool to a living knowledge system. Competitors that offer this create lock-in through the integration layer. Once a customer connects their document pipeline, switching costs become very high.

**Revenue impact**: **Very High** (but long-term) — This is the difference between a tool and a platform. The implementation is complex (webhook integrations, change detection, incremental re-indexing) but creates lasting competitive advantage.

---

#### 8. Custom Evaluation Rubrics

**What it is**: Let customers define their own quality evaluation criteria beyond the default 5 metrics. Example: a legal firm wants "regulatory accuracy" and "precedent citation" as scored dimensions, not just generic "faithfulness."

**Why it's strategic**: This makes the quality layer extensible and domain-specific. It's the kind of feature that enterprise procurement teams care about deeply — it shows the product adapts to their workflow rather than forcing a generic framework.

---

#### 9. Multi-Model Response Ensemble

**What it is**: Query a document using multiple LoRA models simultaneously and let the system (or the user) select the best response. The quality evaluation layer scores each model's response, and over time, the system learns which model performs best for which types of questions.

**Why it's strategic**: This positions BrightRun as a "model orchestration" platform, not just a "fine-tuning tool." It creates a natural upsell path (train more models, deploy more endpoints, get better responses) and generates unique training signal that improves over time.

## Debugging Reference

### LoRA Response is Empty or Truncated
- Check RunPod pod is running and vLLM server started
- Verify `max_tokens` in inference configuration (default 512 may be too low for context-heavy prompts)
- Check `INFERENCE_API_URL_ADAPTED` environment variable is set and reachable

### Model Dropdown Shows No Models
- Check `pipeline_inference_endpoints` table for records with `endpoint_type = 'adapted'` AND `status = 'ready'`
- If join query fails, the endpoint falls back to a simpler query (check server logs for "fallback" message)

### LoRA Response Doesn't Follow Context
- Expected — LoRA models fine-tuned on style/format may not strictly follow document context
- Compare self-eval scores between RAG Only and RAG+LoRA to quantify the gap
- Consider LoRA training with RAG-style data (context + question + answer format) for better grounding

### Self-Eval is Null for LoRA Only Mode
- This is intentional — there's no document context to evaluate against
- Self-eval only runs when RAG retrieval provides context (RAG Only and RAG+LoRA modes)

### Cold Start Latency on Serverless
- First request to a cold worker takes 30-120 seconds
- Use RunPod "Active Workers" setting to keep 1 worker warm (increases cost)
- Consider switching to Pods for lower-latency testing
