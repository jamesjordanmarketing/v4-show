# RAG Frontier Testing Tutorial

**Version:** 1.0  
**Date:** February 11, 2026  
**Purpose:** Step-by-step instructions to test RAG document upload, chat, and identify missing functionality

---

## Prerequisites

1. Dev server running: `cd src && npm run dev`
2. Supabase database with RAG schema (E01-E06 complete)
3. Environment variables:
   - `ANTHROPIC_API_KEY` — For Claude (RAG processing & chat)
   - `OPENAI_API_KEY` — For embeddings
   - `SUPABASE_URL` and `SUPABASE_SERVICE_ROLE_KEY`

## Part 1: Upload a Document to RAG

### Step 1: Navigate to RAG Frontier
1. Start dev server: `cd src && npm run dev`
2. Open browser: `http://localhost:3000`
3. Sign in
4. Go to `/dashboard`
5. Click **"RAG Frontier"** button (teal, with book icon)

### Step 2: Create Knowledge Base
1. Click **"Create Knowledge Base"**
2. Enter:
   - Name: `Test KB`
   - Description: `Testing RAG document upload`
3. Click **"Create"**
4. Select the newly created knowledge base from the list

### Step 3: Upload Document
1. In the **"Upload Document"** panel (left side):
   - Click file picker or drag & drop
   - Select a PDF, DOCX, or TXT file (< 50 MB)
   - Optionally add description
2. Click **"Upload"**
3. Watch status change:
   - `uploading` → `processing` (Claude extracts sections/facts)
   - Takes 1-5 minutes depending on document size

### Step 4: Answer Expert Questions
1. When status becomes `awaiting_questions`:
   - Click the document in the list
   - Go to **"Expert Q&A"** tab (has orange indicator)
   - Answer 3-8 questions generated by Claude
   - Click **"Submit Answer"** for each
   - OR click **"Skip"** if question doesn't apply
2. After answering all questions, click **"Verify Document"**
3. Status changes to `ready`

### Step 5: Chat with Document
1. Go to **"Chat"** tab
2. Select **"RAG Only"** mode
3. Type a question related to the document content
4. Press **Enter** to send
5. Claude retrieves relevant sections/facts and generates a cited answer
6. Citations appear as links (hover to see excerpt)

## Part 2: Push RAG to a LoRA Model (NOT IMPLEMENTED)

### Current Limitation ⚠️

The RAG system **cannot** use trained LoRA models yet. The mode selector shows:
- **RAG Only** ✅ Works (uses Claude + document retrieval)
- **LoRA Only** ❌ Not implemented
- **RAG + LoRA** ❌ Not implemented

**What's Missing:**
1. `queryRAG()` function in `src/lib/rag/services/rag-retrieval-service.ts` ignores the `mode` parameter
2. No integration with `inference-service.ts` (LoRA inference layer)
3. No UI to select which trained model to use for RAG+LoRA queries
4. No prompt construction to combine RAG context with LoRA fine-tuned style

**Workaround:** Use separate systems:
- Use RAG for document-grounded Q&A (Chat tab)
- Use trained models separately via `/models` page (download & deploy manually)

## Part 3: Deploy LoRA Model to RunPod (Separate from RAG)

### Step 1: Train a Model
1. Go to `/datasets` — create or select a dataset
2. Go to `/pipeline/configure` — start a training job
3. Wait for job to complete (shows in `/pipeline/jobs`)

### Step 2: Deploy Trained Adapter
1. Go to `/pipeline/jobs` and click completed job
2. Click **"Deploy to Endpoints"** button
3. System creates two RunPod Serverless endpoints:
   - **Control** (base model only)
   - **Adapted** (base model + your LoRA adapter)
4. Wait 2-5 minutes for endpoints to become `running`

### Step 3: Test LoRA Inference
1. In job detail page, go to **"Test"** tab
2. Enter a prompt
3. Get responses from both Control and Adapted endpoints
4. Compare outputs to see LoRA effect

**Note:** This is completely separate from RAG. You cannot combine RAG retrieval with LoRA-adapted responses in the UI.

## Part 4: Quality Dashboard

### View RAG Quality Metrics
1. After chatting with a document, go to document detail page
2. Click **"Quality"** button (top-right)
3. See metrics:
   - **Faithfulness** — Answer supported by context?
   - **Answer Relevance** — Answer addresses the question?
   - **Context Relevance** — Retrieved sections/facts relevant?
   - **Answer Completeness** — Answer thorough?
   - **Citation Accuracy** — Citations match context?
4. Scores use Claude to self-evaluate each query/response

## Functionality Omissions

### 1. RAG + LoRA Integration (Critical)

**Missing:** Ability to query documents using a fine-tuned LoRA model

**Impact:** Users cannot apply their trained writing style/knowledge to RAG-retrieved context

**What Would Be Needed:**
- UI to select a trained model artifact for RAG queries
- Service layer to:
  1. Retrieve relevant context (already works)
  2. Send context + query to LoRA endpoint instead of Claude
  3. Parse LoRA response and extract citations
- Update `queryRAG()` to route to inference service when `mode !== 'rag_only'`

**Files to Modify:**
- `src/lib/rag/services/rag-retrieval-service.ts` (line 273-402, `queryRAG()` function)
- `src/components/rag/RAGChat.tsx` (add model selector dropdown)
- `src/hooks/useRAGChat.ts` (pass modelId to API)
- `src/app/api/rag/query/route.ts` (accept modelId parameter)

---

### 2. Knowledge Base Sharing

**Missing:** Ability to share knowledge bases across users or teams

**Impact:** Each user's knowledge bases are private

**What Would Be Needed:**
- `rag_knowledge_base_permissions` table
- UI to invite users or set public/private
- Service layer permission checks

---

### 3. Bulk Document Upload

**Missing:** Upload multiple documents at once to a knowledge base

**Impact:** Users must upload documents one by one

**What Would Be Needed:**
- Accept multiple files in `DocumentUploader.tsx`
- API endpoint to handle batch uploads
- Background processing queue

---

### 4. Document Versioning

**Missing:** Upload a new version of an existing document

**Impact:** Users must delete and re-upload to update document content

**What Would Be Needed:**
- `version` column in `rag_documents` table
- Archive old versions instead of deleting
- UI to view version history

---

### 5. Cross-Document Search

**Missing:** Query across all documents in a knowledge base simultaneously

**Impact:** Users can only chat with one document at a time

**Current Workaround:** RAG API accepts `knowledgeBaseId` without `documentId`, but UI doesn't expose this

**What Would Be Needed:**
- Add "Chat with entire knowledge base" button in `/rag` page
- Show which document each citation came from in multi-doc chat

---

### 6. Export Chat History

**Missing:** Download chat transcripts as PDF/JSON

**Impact:** Users cannot save their Q&A sessions

**What Would Be Needed:**
- Export button in chat interface
- API endpoint to generate formatted transcript
- Option to include citations and sources

---

### 7. Auto-Verify Documents

**Missing:** Option to skip Expert Q&A and auto-verify

**Impact:** All documents require manual expert input before chat becomes available

**What Would Be Needed:**
- Checkbox "Skip expert verification" in upload flow
- Change status directly from `processing` → `ready`
- Use case: trusted documents that don't need Q&A grounding

---

### 8. Semantic Search UI

**Missing:** Browse/search documents by semantic similarity to a query

**Impact:** Users must remember document names to find content

**What Would Be Needed:**
- Search bar in knowledge base view
- Display top N documents ranked by relevance
- Click to open document detail

---

### 9. LoRA Model Selection in RAG Chat

**Missing:** Dropdown to choose which trained model to use for LoRA/RAG+LoRA mode

**Impact:** Even if RAG+LoRA were implemented, no way to select the model

**What Would Be Needed:**
- Fetch user's trained models from `/api/models`
- Dropdown in `RAGChat.tsx` (only shown when mode is `lora_only` or `rag_and_lora`)
- Pass `modelId` to `queryRAG()` service

---

### 10. Deployment Status in Models Page

**Missing:** Show which models are deployed to RunPod endpoints

**Impact:** Users don't know if a model is available for inference without checking job page

**What Would Be Needed:**
- Join `model_artifacts` with `inference_endpoints` table
- Badge/icon in `/models` page showing "Deployed" status
- Link to test endpoint directly from model card

## Summary: What Works vs. What Doesn't

### ✅ Working Features

1. **RAG Document Upload** — Upload PDF/DOCX/TXT, extract text
2. **Document Processing** — Claude reads document, generates sections/facts/questions
3. **Expert Q&A** — Answer questions to verify document understanding
4. **RAG Chat** — Query documents with semantic search + Claude response generation
5. **Quality Metrics** — Self-evaluation of query/response quality
6. **Knowledge Base Management** — Create, delete, organize documents
7. **LoRA Training** — Train models on datasets (separate from RAG)
8. **LoRA Deployment** — Deploy trained adapters to RunPod serverless endpoints
9. **LoRA Inference** — Test deployed models (separate from RAG)

### ❌ Missing Features (Critical)

1. **RAG + LoRA Integration** — Cannot use trained models with RAG retrieval
2. **Model Selection in RAG** — No UI to choose which model for LoRA mode

### ❌ Missing Features (Nice-to-Have)

3. Knowledge base sharing
4. Bulk document upload
5. Document versioning
6. Cross-document search (backend exists, UI missing)
7. Export chat history
8. Auto-verify documents
9. Semantic document search UI
10. Deployment status in models page

## Debugging Tips

### Document Stuck in "Processing"
- Check server logs for Claude API errors
- Verify `ANTHROPIC_API_KEY` is valid
- Document may be too long (> 500 pages)

### No Embeddings Generated
- Check `OPENAI_API_KEY` is valid
- Inspect `rag_embeddings` table in Supabase

### Chat Returns Generic "No relevant information" Message
- Check embedding search threshold in `src/lib/rag/config.ts`
- Verify embeddings exist: `SELECT COUNT(*) FROM rag_embeddings WHERE source_type = 'section'`

### LoRA Mode Does Nothing
- Expected behavior — not implemented
- Check code: `src/lib/rag/services/rag-retrieval-service.ts:284` — mode parameter is logged but not used
