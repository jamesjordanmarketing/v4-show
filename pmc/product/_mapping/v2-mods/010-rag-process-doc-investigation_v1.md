When you use the RAG chat in **"RAG Only"** mode (the default), the responses are generated by **Claude** — specifically, the fast Haiku variant.

**How it works (RAG Only flow):**

```
User asks question on /rag/[id]
  → RAGChat.tsx sends POST /api/rag/query with mode='rag_only'
    → queryRAG() in rag-retrieval-service.ts
      → Step 1: Get document summary
      → Step 2: Generate HyDE hypothetical answer (Claude Haiku)
      → Step 3: Embed query + HyDE via OpenAI text-embedding-3-small

      → Step 6: Send context + question to Claude Haiku for response
      → Step 7: Self-evaluate with Claude Haiku

```

**Key point:** In RAG-only mode, **no LoRA adapter is involved at all**. The entire pipeline is Claude Haiku + OpenAI embeddings. This is why the responses in RAG-only mode have no "emotional" or "supportive" language — Claude Haiku is a general-purpose model, and the system prompt specifically instructs it to be factual and citation-based:

> *"You are a helpful knowledge assistant that answers questions based on provided source material... Only use information from the provided context... Never fabricate information not in the context"*

## Question 3: Can RunPod Workers Cache or Retain Context from Previous Questions?

### Answer: **No. Each request is stateless and independent.**

**Evidence from `inference-pods.ts`:**

```typescript
// Each request is a fresh POST to /v1/chat/completions
const requestBody = {
    model,          // model identifier
    messages,       // only current system prompt + user message
    max_tokens: 2048,
    temperature: 0.7,
};
```

**Key observations:**

1. **No conversation history is sent.** Each API call includes only the current `systemPrompt` and `prompt` — there is no `messages` array containing prior turns. (See `inference-pods.ts`, lines 83-87.)

2. **vLLM is stateless.** The RunPod Pods run vLLM with an OpenAI-compatible API. vLLM does not maintain session state between requests. Each `/v1/chat/completions` call is independent.

3. **No `session_id` or `thread_id` is passed.** There is no mechanism in the code to link consecutive requests.

4. **No KV-cache persistence.** vLLM's KV-cache is internal to each request and is released when the request completes. There is no cross-request caching.

> [!NOTE]
> The fact that the standalone LoRA (`pipeline/jobs/[id]`) and the RAG+LoRA (`/rag/[id]`) produce different tones for the same question is **not** due to caching — it's due to the different system prompts used by each interface.

## Question 5: How Does the RAG System Embed Data? Detailed Analysis

### 5.1 The Embedding Process — Step by Step

The RAG document processing pipeline has **three distinct phases**. Here is exactly what happens:

#### Phase 1: Claude Reads the Document (Frontier Model Analysis)

**File:** `claude-llm-provider.ts`, `readDocument()` method (lines 110-290)

Claude Haiku 4.5 receives the **entire document text** and a detailed system prompt. It outputs a structured JSON with:

| Output | Count | Description |
|--------|-------|-------------|
| `summary` | 1 | 300-500 word document summary |
| `sections` | 5-15 | Major sections with full original text + summaries |
| `facts` | 50-150 | Atomic factual statements (the core of the RAG knowledge) |
| `entities` | 10-30 | Key people, organizations, concepts |
| `topicTaxonomy` | 3-8 | Topic tags |
| `ambiguities` | 3-10 | Unclear/ambiguous items |
| `expertQuestions` | 5 | Questions for domain expert review |

**Each fact has:**
```json
{
    "factType": "fact|entity|definition|relationship|table_row|policy_exception|policy_rule",
    "content": "The atomic factual statement",
    "sourceText": "Brief excerpt showing source",
    "confidence": 0.95
}
```

> [!IMPORTANT]
> **Claude is NOT querying the document for missing information.** It is performing *knowledge extraction* — decomposing the document into atomic, searchable facts. It reads the document once and extracts everything it can identify as a discrete factual claim, definition, rule, exception, or table entry.

#### Phase 2: Facts Are Embedded via OpenAI

**File:** `src/lib/rag/services/rag-embedding-service.ts` (based on config)
**Model:** OpenAI `text-embedding-3-small` (1536 dimensions)

Each extracted fact, section, and the document summary is converted into a **1536-dimensional vector** using OpenAI's embedding model. These vectors are stored in the `rag_embeddings` table with a `tier` classification:

| Tier | Content | Purpose |
|------|---------|---------|
| Tier 1 | Document summary | Broad document-level matching |
| Tier 2 | Section texts | Section-level retrieval |
| Tier 3 | Individual facts | Fine-grained factual retrieval |

#### Phase 3: Storage in Database

Everything is stored in Supabase:
- `rag_documents` — document metadata + summary
- `rag_sections` — extracted sections with original text
- `rag_facts` — extracted facts with content, type, confidence
- `rag_embeddings` — vector embeddings for similarity search

### 5.2 What Are the Expert Questions?

The expert questions are **NOT used during embedding or retrieval.** They serve a separate purpose:

**Source:** `claude-llm-provider.ts`, lines 165-170

```json
"expertQuestions": [
    {
        "questionText": "The question to ask the domain expert",
        "questionReason": "Why this matters",
        "impactLevel": "high|medium|low"
    }
]
```

Claude generates exactly 5 high-impact questions about things that are **ambiguous, unclear, or potentially incomplete** in the document. These appear in the "Expert Q&A" tab on `/rag/[id]` and allow a human domain expert to answer them. If answered, the answers are fed back to Claude via `refineKnowledge()` to update section summaries and add new facts.

**The purpose of expert questions:** They allow Claude to flag areas where the document is unclear and a human expert's input would improve the knowledge base quality. For example, if a policy cross-references another policy that isn't in the document, Claude might ask "What is the specific procedure for BC-ELIG-004 (Account Closure)?"

## 8. How to Distinguish "RAG Missing Data" vs. "Model Hallucination"

### The Core Problem

When the model gives a wrong answer, there are **three possible root causes**:

```
Source Document → Claude Extraction → Embedding → Retrieval → Model Response
                    ↑                    ↑            ↑           ↑
                  Gap #1              Gap #2       Gap #3      Gap #4
                 (Extraction)        (Embedding)  (Retrieval) (Generation)
```

### Gap Analysis Framework

| Gap | Question | How to Test |
|-----|----------|-------------|
| **Gap #1: Extraction** | Did Claude extract this fact from the document? | Query `rag_facts` for the specific information |
| **Gap #2: Embedding** | Was the fact successfully embedded? | Compare `rag_facts` count vs `rag_embeddings` Tier 3 count |
| **Gap #3: Retrieval** | Was the fact retrieved for this specific query? | Enable debug logging in `queryRAG()` to log retrieved fact IDs |
| **Gap #4: Generation** | The model had the fact but misinterpreted or ignored it | Check the assembled context sent to the model |

### Diagnostic Steps for a Specific Wrong Answer

**Step 1: Check if the fact exists in the database**

```sql
SELECT * FROM rag_facts 
WHERE document_id = 'ceff906e-968a-416f-90a6-df2422519d2b'
  AND content ILIKE '%high liquidity%';
```

- If **no rows returned** → **Gap #1** (Claude didn't extract it)  
- If rows returned → proceed to Step 2

**Step 2: Check if the fact has an embedding**

```sql
SELECT e.id, e.tier, e.source_type, f.content 
FROM rag_embeddings e 
JOIN rag_facts f ON e.source_id = f.id 
WHERE f.document_id = 'ceff906e-968a-416f-90a6-df2422519d2b'
  AND f.content ILIKE '%high liquidity%';
```

- If **no embedding** → **Gap #2** (OpenAI embedding failed)
- If embedding exists → proceed to Step 3

**Step 3: Check if the fact was retrieved**

Currently, the retrieval service does NOT log which specific facts were retrieved. Adding a debug log line would help:

```typescript
// In rag-retrieval-service.ts, after assembleContext()
console.log(`[RAG Debug] Retrieved facts: ${facts.map(f => f.id).join(', ')}`);
console.log(`[RAG Debug] Assembled context length: ${assembledContext.length} chars`);
```

**Without code changes**, you can infer retrieval by examining the Vercel log `promptLength`. If the context is large (~26K chars as seen in log-22), the retrieval is working — the question is whether the *specific* fact was included.

**Step 4: Check if it's a generation error**

If the fact exists, was embedded, and was retrieved (included in the context), but the model still gave a wrong answer → **Gap #4** (Model hallucination). This means the model had the information available but:
- Lost it in a long context (attention limitation)
- Confused it with other information
- Was overridden by the system prompt's constraints
- The LoRA's fine-tuning interfered with factual grounding

```typescript
    of institutional policies. Use the provided context to answer accurately, but maintain 
    your natural supportive and warm communication style. Include citations in brackets 
    like [Section: title] when referencing specific information. If information is not 
    found in the context, say so honestly rather than guessing.`;
    prompt = `Context from knowledge base:\n\n${assembledContext}\n\n---\n\n
    Question: ${queryText}\n\nAnswer using the context above, maintaining your natural tone:`;
}
```

### What Changed and Why

| Aspect | Before | After | Reasoning |
|--------|--------|-------|-----------|
| Persona | "helpful knowledge assistant" | "helpful, empathetic financial assistant" | Matches LoRA's fine-tuned personality |
| Context constraint | "using ONLY the provided context" | "Use the provided context to answer accurately" | Removes the absolute constraint that was overriding LoRA personality |
| Tone instruction | None | "maintain your natural supportive and warm communication style" | Explicitly allows LoRA's trained tone to express itself |
| Unknown handling | None | "If information is not found in the context, say so honestly" | Prevents confabulation (the High Liquidity Offset error) |
| Prompt ending | "Answer using only the context above:" | "Answer using the context above, maintaining your natural tone:" | Reinforces personality preservation |
