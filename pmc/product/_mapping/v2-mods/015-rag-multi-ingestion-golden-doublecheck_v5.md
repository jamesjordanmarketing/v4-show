## Q4: Does Anything Need Tweaking or Fixing for Reliability/Quality?

### Priority 1 — Bug A: Fix `fact_type` validation in Pass 6 verification (Medium effort, High impact)

The Opus verification pass generates recovery facts with `fact_type` values that don't exist in the database enum. This causes ~13 facts per run to be silently rejected.

**Root Cause**: The Pass 6 prompt in `src/lib/rag/providers/claude-llm-provider.ts` instructs Claude to set `factCategory` to `"verification_recovery"` for all recovery facts, but does **not** constrain `factType` to the 14 allowed values. Claude invents types that make semantic sense (e.g., `scope`, `reference`, `date_value`) but are not in the allowed set.

**The 14 allowed `fact_type` values** (from `src/types/rag.ts` and `rag_facts_fact_type_check` constraint):
```
"fact" | "entity" | "definition" | "relationship" | "table_row" |
"policy_exception" | "policy_rule" | "limit" | "threshold" |
"required_document" | "escalation_path" | "audit_field" |
"cross_reference" | "narrative_fact"
```

**Recommended fix**: Two-layered approach:

#### Fix A1 — Prompt Constraint (`src/lib/rag/providers/claude-llm-provider.ts`)

In the `verifyExtractionCompleteness` method, find the RULES section of the Pass 6 system prompt and add an explicit enumeration of allowed `factType` values.

**Current prompt rule (approximate):**
```typescript
// Rule 2: Set factCategory to "verification_recovery" for all facts found by this pass
```

**Updated prompt rule:**
```typescript
// Rule 2: Set factCategory to "verification_recovery" for all facts found by this pass
// Rule 3: factType MUST be one of these exact strings (choose the closest match, use "fact" if unsure):
//   "fact", "entity", "definition", "relationship", "table_row",
//   "policy_exception", "policy_rule", "limit", "threshold",
//   "required_document", "escalation_path", "audit_field",
//   "cross_reference", "narrative_fact"
```

#### Fix A2 — Normalization Fallback (in `storeExtractedFacts`, `src/lib/rag/services/rag-ingestion-service.ts`)

Add a safety-net validation step before the DB insert to catch any `factType` values that slip through despite the prompt constraint:

```typescript
const ALLOWED_RAG_FACT_TYPES = new Set([
    'fact', 'entity', 'definition', 'relationship', 'table_row',
    'policy_exception', 'policy_rule', 'limit', 'threshold',
    'required_document', 'escalation_path', 'audit_field',
    'cross_reference', 'narrative_fact',
]);

// When building the DB row insert object for each fact:
fact_type: ALLOWED_RAG_FACT_TYPES.has(fact.factType) ? fact.factType : 'fact',
```

This ensures no `rag_facts_fact_type_check` constraint violations can occur at insert time, regardless of what the LLM produces.

### Priority 2 — Bug B: Fix empty table content guard in Pass 3 (Low effort, Low impact)

One table at line 1126 produced a parse failure because Claude was sent empty content between `[TABLE]` markers. The table had no actual content. This wasted an API call and produced an error log entry, though the pipeline continued correctly.

**Root Cause**: In `src/inngest/functions/process-rag-document.ts`, the Pass 3 loop calls `provider.extractTableData()` for every table in `structure.tables` without first checking whether `table.content` is non-empty.

**Fix B — Pre-call empty content guard (`src/inngest/functions/process-rag-document.ts`)**

Inside the Pass 3 per-table `step.run()`, before calling `provider.extractTableData()`, add:

```typescript
const rawContent = table.content?.trim() ?? '';
if (rawContent.length < 10) {
    console.log(
        `[Inngest] Pass 3: Table ${tableIdx} (${table.tableName ?? 'unnamed'}) — ` +
        `empty or too-short content (${rawContent.length} chars), skipping LLM call`
    );
    return [] as FactExtraction[];
}
```

The threshold of `< 10` chars catches genuinely empty tables and near-empty placeholders while allowing any table with real content (even very small ones) to proceed.
