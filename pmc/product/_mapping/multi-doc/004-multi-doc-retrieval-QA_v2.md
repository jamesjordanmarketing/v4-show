## 3. Root Causes (3 Issues)

### RC-A: classifyQuery crashes 100% — enhanced prompt causes systematic JSON parse failure

**Severity: CRITICAL**  
**Impact:** The multi-hop/comparative branch is never taken. All queries default to "simple" standard path.

The enhanced classifyQuery prompt (added in v1 fixes) contains:
1. A much longer system prompt (~600 tokens vs ~150 before)
2. Unicode arrow characters (`→`) in examples
3. More detailed instructions that may cause Haiku to produce preamble text before JSON

The `catch` block silently swallows the error:
```typescript
} catch {
    console.warn('[RAG Retrieval] Query classification failed, defaulting to simple');
    return { type: 'simple' };
}
```

**Fix needed:**
1. Log the actual error: `catch (err) { console.warn('...failed:', err); ... }`
2. Replace `→` arrows with `=` or `-->` to avoid Unicode issues
3. Add more aggressive JSON extraction: strip everything before the first `{` and after the last `}`
4. Consider increasing `maxTokens` from 300 to 500
5. Add explicit "RESPOND WITH ONLY THE JSON OBJECT. NO OTHER TEXT." at the end of the prompt

### RC-B: LoRA mode (`rag_and_lora`) completely bypasses multi-doc generation prompt

**Severity: HIGH**  
**Impact:** Any query using LoRA+RAG mode will NEVER have multi-doc awareness.

In `generateLoRAResponse()` (line 738), the system prompt is hardcoded:
```typescript
systemPrompt = `You are a helpful, empathetic financial assistant with deep knowledge 
of institutional policies. Use the provided context to answer accurately, but maintain 
your natural supportive and warm communication style...`;
```

This prompt has:
- ❌ No multi-doc completeness instruction
- ❌ No document enumeration
- ❌ No instruction to address all documents
- ❌ No awareness that the context contains `## From:` headers from multiple sources

The LoRA model receives the full assembled context (including `## From:` headers and interleaved sections) but has zero guidance about addressing all documents.

**Fix needed:**
1. Add multi-doc detection to `generateLoRAResponse()`: check if `assembledContext.includes('## From:')`
2. When multi-doc detected, inject a multi-doc instruction into the LoRA system prompt
3. Extract document names from headers (same regex as `generateResponse`) and list them explicitly

### RC-C: generateResponse multi-doc prompt not strong enough for imbalanced content

**Severity: MEDIUM**  
**Impact:** When the section ratio is heavily skewed (e.g., 9:3) AND the question aligns much more strongly with one document's domain, Claude may still produce a single-doc answer despite the multi-doc instruction.

The Q1 margin call question succeeded (0.92) because both banks had directly comparable lending products. The Q3 Tuscany lockout question failed because:
- Sun Bank's FIDO2/Geographic Anomaly Mode policies directly answer the user's specific complaints
- Moon Bank's Slate/Biometric Vein system is a fundamentally different architecture that doesn't have a direct "locked out with FIDO2 key" equivalent
- Claude likely determined Moon Bank's authentication policies aren't relevant to the user's specific situation (which is arguably a correct judgment for a single-user query)

**Fix needed (defense in depth):**
1. Strengthen the multi-doc prompt's language: instead of "address EACH document's relevant information" → "ALWAYS present the equivalent policies from EACH document, even if they approach the topic differently"
2. Add explicit fallback instruction: "If a document handles this topic differently than the user describes, explain how that document's approach differs"
3. Consider adding a "comparative summary" suffix instruction: "End your response with a brief comparison noting the key differences between the documents"

## 6. Why Q1 Succeeded but Q3 Failed (Same Fixes Active)

| Factor | Q1 (Margin Call) ✅ | Q3 (Tuscany Lockout) ❌ |
|---|---|---|
| **Content balance** | 9 Sun : 5 Moon (64:36) | 9 Sun : 3 Moon (75:25) |
| **Topical overlap** | BOTH banks have explicit lending/LTV/margin call policies | Sun Bank has very specific FIDO2/lockout policies; Moon Bank's Slate system is architecturally different |
| **Semantic alignment** | "LTV ratio" matches both banks equally | "FIDO2 key", "authenticator app" only appear in Sun Bank |
| **Response generation** | Claude's multi-doc prompt sufficient | Claude's multi-doc prompt insufficient for this content skew |
| **Classification** | Failed → simple (but prompt fix compensated) | Failed → simple (prompt fix couldn't compensate) |

The key insight: **the `generateResponse` multi-doc prompt is a necessary but not sufficient fix**. It works when both documents have comparable content for the question. It fails when one document's content is overwhelmingly more specific/relevant to the exact terms used in the question.

This is why the `classifyQuery` fix is CRITICAL — when properly classified as "comparative," the multi-hop branch runs **separate sub-queries per document**, ensuring each document gets its own focused retrieval pass. The standard path retrieves all content in one pass where one document can dominate the results.

## 7. Architecture Observation

The pattern across v1 and v2 testing reveals a **defense-in-depth gap**:

```
Layer 1: classifyQuery     → BROKEN (crashes, defaults to simple)
Layer 2: balanceMultiDoc   → WEAK (60% cap allows 75:25 ratios)
Layer 3: assembleContext    → FIXED ✅ (interleaving working)
Layer 4: generateResponse   → PARTIALLY EFFECTIVE (works for balanced content, fails for skewed)
Layer 5: generateLoRAResponse → MISSING (no multi-doc awareness at all)
Layer 6: selfEvaluate       → FIXED ✅ (correctly catches single-doc answers)
```

The system needs at least 2 of layers 1-5 working correctly to consistently produce multi-doc answers. Currently only Layer 3 (interleaving) is fully working. Fixing Layer 1 (classifyQuery) and Layer 5 (LoRA prompt) will restore the required redundancy.
