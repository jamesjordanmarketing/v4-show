## Section A — Drop `kv_store_*` Tables

### What are they?

The 35 `kv_store_*` tables were auto-generated by **Figma Make** — a Supabase plugin used during very early prototyping. Each table has this schema:

```sql
CREATE TABLE kv_store_<hash> (
  key  TEXT NOT NULL PRIMARY KEY,
  value JSONB NOT NULL
);
```

Each unique Figma Make session creates its own `kv_store_<random-hash>` table. They are Figma-side key-value stores for UI state during prototyping.
**James' Decision**: I agree with this.

**No.** The only code file that references `kv_store` is `src/supabase/functions/server/kv_store.tsx`. That file is a **Supabase Edge Function** (server-side Deno runtime, not the Next.js app). It references `kv_store_6b04c628` — which is NOT one of the 35 tables in the database (it was a different session's table, likely already deleted). The file is autogenerated by Figma Make and is not called by any Next.js code.

There are **zero references** to `kv_store` in any file under `src/app/`, `src/lib/`, `src/components/`, or `src/hooks/`.

### Impact of dropping them?

**Zero impact.** They are completely isolated from the application.

### Combined size?

## Section B — The "Dangling Ownership" Problem (Explained & Resolved)

### What is it?

In the database, rows in `conversations`, `batch_jobs`, and `documents` have a `created_by` UUID that points to a user account that no longer exists in `auth.users`. This happened because:

1. Early in development you created user accounts to test with
2. You deleted those test accounts (or they were cleaned up)
3. The rows those accounts created remained in the database
4. When E01's backfill tried to copy `created_by → user_id`, it failed with a FK violation because `user_id` has a FK constraint to `auth.users`

### What data is affected?

| Table | Dangling records | Type of data |
|-------|-----------------|--------------|
| `conversations` | 418 | Test conversations (raw + enriched) |
| `batch_jobs` | 55 | Test batch generation jobs |
| `documents` | 2 | Test documents (chunks workflow) |

All 513 dangling records are logged in `_orphaned_records` with `notes = 'dangling_created_by: user deleted from auth.users'`.

### The Resolution Strategy

**Delete, not reassign.** Since all of this is test-era data with no production value, the cleanest resolution is deletion. The global reset script in Section E will delete all user-generated activity, including these 513 records. After Section E runs:

- `conversations` will be empty → dangling records gone
- `batch_jobs` will be empty → dangling records gone
- `documents` (chunks module) will be empty → dangling records gone

### What about `_orphaned_records`?

After the global reset, truncate `_orphaned_records`:

After the global reset:
- All tables targeted by E04's NOT NULL constraints will have zero rows
- Applying `ALTER TABLE x ALTER COLUMN user_id SET NOT NULL` on an empty table always succeeds
- As data is written by E02–E03's secured routes, `user_id` is set at insert time
- E04 will proceed without issue

**No orphan user account is needed. No data reassignment is needed. The clean path is full deletion of test data.**
